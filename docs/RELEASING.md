# SNAPLINK Release Guideline

본 문서는 SNAPLINK 프로젝트의 일관된 배포 프로세스와 버전 관리 표준을 정의합니다.

---

## 📌 1. 버전 관리 표준 (SemVer)

SNAPLINK는 유의적 버전 관리(Semantic Versioning)를 준수합니다.
형식: `Major.Minor.Patch` (예: `1.0.0`)

| 버전 유형 | 예시 | 변경 기준 |
| :--- | :--- | :--- |
| **Major** | `1.x.x` → `2.0.0` | 하위 호환성이 깨지는 대규모 변경 또는 리브랜딩 |
| **Minor** | `1.0.x` → `1.1.0` | 새로운 기능 추가, 네이티브 모듈 변경 (`pod install` 필요 시) |
| **Patch** | `1.0.0` → `1.0.1` | 버그 수정, 네이티브 코드의 미세 수정, 스토어 재빌드 베이스라인 갱신 |

---

## ⚡ 2. 배포 전략

### 2.1 OTA 업데이트 (Revopush)
네이티브 변경이 없는 단순 JS logic, Style, Asset 수정 시 사용합니다.

- **기준:** 현재 바이너리 타겟 버전(`-t`)을 유지하며 배포합니다. 
- **버전 표기:** 앱 버전을 올리지 않고 Revopush 내부 Label(`v1`, `v2`...)을 통해 히스토리를 관리합니다.
- **명령 예시:** `revopush upload --app snaplink/ios -t "1.0.0" ...`

### 2.2 스토어 재빌드 (Native Update)
아래 상황에서는 반드시 버전을 올리고 스토어 심사를 받아야 합니다.

- **네이티브 라이브러리 추가/삭제:** 새로운 npm 패키지 설치 후 `pod install`이 필요한 경우
- **네이티브 코드 수정:** `ios/` 또는 `android/` 폴더 내 파일 변경 시
- **배포 롤업(Roll-up):** OTA 업데이트가 누적되어 데이터량이 많아진 경우(약 10~20회), 신규 유입자 편의를 위해 베이스라인을 갱신합니다.

---

## 🚀 3. 배포 워크플로우

### 1단계: Staging (검증)
- 모든 수정 사항은 먼저 Staging 환경으로 배포합니다.
- iOS의 경우 TestFlight를 통해 내부 테스터 및 개발자가 실제 기기에서 검증합니다.

### 2단계: Production (출시)
- Staging에서 검증이 완료된 빌드는 `promote` 명령어를 사용하여 Production으로 즉시 배포합니다.
- 사용자에게 배포하기 전 최종 QA 단계를 거칩니다.

### 3단계: Store 출시 후 관리
- 스토어 심사가 완료되어 신규 버전이 출시되면, 해당 버전에 맞는 새로운 배포 키와 타겟 버전을 설정하여 이후의 OTA 업데이트를 준비합니다.

---

## 🔔 CI/CD 연동
배포 프로세스는 GitHub Actions를 통해 자동화됩니다.
- `main` 브랜치 병합 시 자동으로 Staging 배포가 트리거될 수 있도록 구성합니다.
- 수동 트리거(Workflow Dispatch)를 통해 Production 배포 및 스토어 제출을 수행합니다.
